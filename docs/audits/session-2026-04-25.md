## 4. Parser wsmart cobre audio/image/video/ExtendedText (commit `9ca6dd6`)

**Mudanças em `lib/uazapi/types.ts`:**
- Strip sufixo `Message` (regex `/Message$/i`) + lowercase pra matching tolerante
- Branch text aceita `Conversation` / `ExtendedText` / `ExtendedTextMessage` (qualquer capitalização)
- Branches novas pra audio/image/video usando `m.type` como sinal forte
- Extração de URL/mimetype defensiva (tenta `m.url`, `m.mediaUrl`, `m.audioMessage.url` — desconhecíamos o shape exato wsmart de mídia)
- Reaction/Sticker/Contact/Document/Poll seguem em `other` (preserva `rawType` original)

**Mudança crítica em persist:**
- `messages.raw_payload` agora armazena o body HTTP cru (não o evento Zod-normalizado). Permite refinar parser depois sem replay de webhook.
- Plumbing: `route.ts` → `handler.ts` → `persist.ts` aceitam `rawBody` como param opcional (fallback p/ event normalizado mantém testes funcionando).

**Limitação documentada:** rows pré-fix têm `raw_payload` normalizado, URLs/mimetype originais dos 173 áudios/imagens **perdidos**. Forward-fix only.

5 tests novos (ExtendedText / Audio / Image / Reaction) + 1 atualizado (degrade pra `other` virou classify pra `audio`). 354 → 355 tests.

## 6. fromMe + audio: id-based skip (commit `877a7f4` + migration 0015)

**Gatilho:** user manda áudio do próprio celular. Persist tem regra:
```ts
if (event.key.fromMe && event.content.kind === "audio") {
  return { status: "ignored", reason: "fromMe audio (own podcast delivery)" };
}
```
Skip cego pra evitar loop quando NÓS entregamos o podcast (também vem `fromMe=true` pelo webhook). Mas isso bloqueia o owner de ter os áudios dele transcritos.

**Solução:** distinguir nossas entregas via `audios.uazapi_delivered_message_id`:

- **Migration 0015** — coluna nova + index parcial
- **`sendAudio`** em `uazapi/client.ts` agora retorna `SendMessageResponse` (era `void`). Resposta carrega o `id` do WhatsApp message
- **`delivery/service.ts::markDelivered`** aceita o id e grava na coluna nova (só quando o destino foi o GRUPO de origem; envios pra "mim"/contato avulso são teste)
- **`webhook/persist.ts`**: quando `fromMe+audio`, faz `SELECT … WHERE uazapi_delivered_message_id = key.id`. Match → skip. No match → processa
- Conservador em erro de DB: skipa pra não retranscrever podcast em loop

Tests: 2 novos cobrindo os 2 caminhos (match=skip vs no-match=process). Mock atualizado pra suportar `whatsapp_instances` + `audios`.

## 7. **BUG BLOQUEADOR DESCOBERTO**: HMAC env var faltando em prod

**Gatilho:** user mandou áudios de teste, nada chegava no DB. Última msg de qualquer grupo era de 2 dias atrás.

**Diagnóstico:** simulei o que n8n faz (assinar body com `79cd5e31...` e POST com `x-podzap-signature`). Resposta:
```
500 "Webhook server misconfigured"
→ x-podzap-signature received but UAZAPI_WEBHOOK_HMAC_SECRET not set
```

**Causa:** env var `UAZAPI_WEBHOOK_HMAC_SECRET` nunca foi adicionada no Portainer Stack. App fail-closed com 500 em **TODA** request com header HMAC. Pipeline morto desde 2026-04-24 23:44 (provavelmente quando o forwarding pelo n8n foi configurado e a env var nunca foi setada).

**Fix:** user adicionou a env var no Portainer + Update the stack. Container recriado, HMAC validando. Probe sintético com curl confirmou:
```
status: 200 | {"ok":true,"status":"persisted","cid":"..."}
```

**Ação follow-up sugerida (não implementada):** secret está exposto no JSON do n8n compartilhado nesta sessão — rotacionar em ambos os lados quando convier.

## 8. Decryption de WhatsApp `.enc` URLs (commit `739dc48`)

**Gatilho:** áudio chega no DB (`type=audio`, `media_url=https://mmg.whatsapp.net/...enc`) mas `media_download_status=failed`. Causa: URL é blob criptografado AES, precisa do `mediaKey` pra descriptografar.

**Solução:** UAZAPI expõe `POST /message/download` que faz a decryption server-side e retorna URL plain hosted no UAZAPI CDN. Verificado live:
```
POST /message/download
body: { id, return_link: true, generate_mp3: false }
→ { fileURL: "https://wsmart.uazapi.com/files/<sha256>.ogg", mimetype: "audio/ogg" }
```
URL retornada baixa direto via fetch (Ogg/Opus 15.7 KB).

**Implementação:**

- **`client.downloadMedia(instanceToken, whatsappMessageId)`** novo em `lib/uazapi/client.ts`
- **`DownloadOpts.uazapiResolve`** novo em `lib/media/download.ts`. Quando URL é WhatsApp `.enc` E opts presentes, downloader chama UAZAPI primeiro pra resolver, depois fetch normal
- **`persist.ts`**: nova helper `loadUazapiResolveOpts` busca + decripta instance token via `lib/crypto`. Plumba pro `downloadAndStore`
- **2 retry workers** (`retry-pending.ts` + `media-download-retry.ts`) também buscam instance token + uazapi_message_id e plumbam
- Tests do retry-workers ganham `whatsapp_instances` table no mock (+ field `uazapi_message_id` em `MessageRow`)
